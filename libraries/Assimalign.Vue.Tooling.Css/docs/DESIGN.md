# Assimalign.Vue.Tooling.Css ‚Äî design

Why this library exists and how it is shaped. It is the **shared, build-time CSS composition core**
introduced with **[V01.01.12.12]** (issue #155) to complete the physical static-web-asset bundling that
`Assimalign.Vue.Syntax.Css`'s `DESIGN.md` recorded as a deferred non-goal (RS1035: a source generator emits
C#, not content files).

## Why a separate library

The `.viu` `@style` compilation ‚Äî parse the component, run the scoped / CSS-Modules / `v-bind()` rewrites,
serialize deterministically ‚Äî must be run by **two build-time hosts**:

1. the `Assimalign.Vue.Syntax.Generators` incremental **source generator**, which emits the `ExtractedStyles`
   constant inside the Roslyn analyzer sandbox (no I/O), and
2. the `VuecsBundleCss` **MSBuild task** (`analyzers/Assimalign.Vue.Tooling.Tasks`), which runs *outside* the
   analyzer sandbox and writes the bundled stylesheet to publish output.

If each host had its own copy of that logic, the generated constant and the physical file could drift. So the
compilation lives **once**, here, and both hosts call it. Running the one deterministic implementation over
the same `.viu` inputs is what makes the bundle **byte-identical** to the generated `ExtractedStyles`
constant ‚Äî the central design guarantee (`docs/UTILITY-CSS-DESIGN.md` ¬ß2.4). The generator was refactored to
delegate to this core (it previously owned the logic in its `Internal/` folder); nothing about the compiled
output changed, which the generator's existing style/caching tests pin.

This is **Tooling / composition-root code, not a peer `Assimalign.Vue.Syntax.*` language library** ‚Äî which is
precisely why it is *allowed* to reference `Assimalign.Vue.Syntax.Css` (and the `.viu`/template parsers) for
emission. A language library may not reference another language library; a composition root may. The
dependency arrows point toward the base and the language libraries, never sideways between them.

## Surface

| Type | Role |
| --- | --- |
| `SingleFileComponentParserFactory` | The shared `.viu` parser composition (the registration seam the generator's composition root used to own). `Create()` registers `@template` + `@style`; `CreateForStyleExtraction()` registers only `@style` ‚Äî byte-identical `@style` results, and it never loads the template compiler / Roslyn, keeping the MSBuild task's load lean. |
| `StyleScopeId` | The `data-v-<hash>` scope-id derivation (FNV-1a over the project-relative `.viu` path). Both hosts resolve the identical id so the scoped CSS matches. |
| `SingleFileComponentStyleCompiler` | The compilation itself: `Compile(parse, scopeId)` (generator, which already holds the parse) and `CompileFile(parser, text, path, projectDirectory)` (task). Returns a `SingleFileComponentStyleCompilation`. |
| `SingleFileComponentStyleBundler` | Composes a project's components into one deterministic bundle string (ordering, LF-only layout). Pure ‚Äî the task performs the file I/O and hands in the already-read text (`SingleFileComponentStyleInput`), so the core stays I/O-free and analyzer-sandbox-safe. |

## Boundaries

- **netstandard2.0** (`$(TargetFrameworkForAnalyzers)`), `EnablePreviewFeatures=false`, linked
  `Shims/` ‚Äî it loads in Roslyn analyzer hosts alongside the rest of the syntax cluster.
- **No I/O.** The core never touches the filesystem; the `VuecsBundleCss` task (outside the sandbox) reads and
  writes. This is how the engine respects RS1035 while still landing a physical stylesheet.
- **No reflection, no dynamic codegen.** Recoverable: a malformed `v-bind()` surfaces as a diagnostic carried
  back to the host (`SingleFileComponentStyleDiagnostic`), never thrown; the only expected exception is
  `OperationCanceledException`.
- **Deterministic, byte-stable output** (LF, ordinal ordering) ‚Äî the incremental-cache and byte-comparison
  contract.

## The bundle layout

`SingleFileComponentStyleBundler` writes:

```
/* <auto-generated> Vuecs single-file-component styles ([V01.01.12.12]). Do not edit ‚Ä¶ */

/* <project-relative path> (<scope-id>) */
<the component's compiled CSS, verbatim>

/* ‚Ä¶ next component ‚Ä¶ */
```

Components are ordered by ascending ordinal of their project-relative path (normalized to forward slashes).
Every newline the bundler emits is `\n`; the compiled CSS segments are LF for scoped / module / `v-bind()`
blocks (the canonical serializer emits `\n`) and preserve source newlines only for verbatim pass-through
blocks ‚Äî matching the generator's constant exactly.

## Non-goals / future

- **Utility-first CSS generation ([V01.01.12.13]+).** The utility engine is the flagship *second* producer of
  opaque CSS strings for the same `VuecsBundleCss` task. The bundler already bundles opaque compiled-CSS text
  with no utility-specific knowledge, so the utility stylesheet plugs in without changing this contract. The
  seam is left; the engine is not built here.
- **Host-page `<link>` injection.** The task registers the bundle as a static web asset with a stable logical
  URL; loading it is a documented explicit `<link rel="stylesheet" href="<AssemblyName>.vuecs.css">`.
  Automatic in-place rewrite of the host page's static web asset is deferred ‚Äî it desyncs the .NET SDK's
  static-web-asset compression/endpoint negotiation graph, so the explicit reference is the shipped path.
